# DuckDB 数据分析功能设计文档

## 一、功能概述

### 1.1 背景

Wegent 项目对 Excel/CSV 文件的原有处理采用**文本提取+智能截断**模式，存在以下问题：

| 问题 | 表现 | 根因 |
|-----|------|------|
| 无结构化查询能力 | AI 只能"阅读"文本，无法执行 SUM/AVG/GROUP BY 等 SQL 操作 | 文本提取丢失数据结构 |
| 知识库 2MB 限制 | 大文件被跳过 RAG 索引 | 文本化后无法有效分块检索 |
| 大文件信息丢失 | 中间数据被采样截断丢弃 | 上下文窗口有限，无法容纳完整数据 |

### 1.2 目标

实现：**AI 自然语言指令 → 自动生成 SQL → 执行结构化查询 → 返回结果**

同时用 DuckDB 的 SUMMARIZE 摘要+样本数据**完全替代**现有 ExcelTruncationStrategy 文本注入。

### 1.3 功能范围

| 场景 | DuckDB 生成 | 触发方式 |
|-----|------------|---------|
| 知识库文档上传 | ✅ 支持 | Celery 异步任务 |
| 对话中上传附件 | ❌ 不支持 | 文本提取（降级） |

## 二、架构设计

### 2.1 整体架构

```text
┌─────────────────────────────────────────────────────────────────────┐
│                         Wegent 架构                                  │
├─────────────────────────────────────────────────────────────────────┤
│                                                                      │
│  Frontend            Backend              knowledge_runtime :8200    │
│   :3000               :8000                                          │
│                         │                       │                    │
│  上传Excel/CSV ─────────┼───────────────────────┤                    │
│       (知识库)          │                       ├── RAG检索（现有）   │
│                         │                       │                    │
│                         │  内部上传端点          ├── DuckDB生成       │
│                         │◄──────────────────────│  (导入+SUMMARIZE   │
│                         │  .duckdb文件           │   +序列化+上传)    │
│                         │                       │                    │
│                         ▼                       └────────────────────┤
│               ┌─────────────────────┐                                │
│               │       MySQL         │                                │
│               │  ├── subtask_contexts                               │
│               │  │     ├── binary_data (Excel/CSV原始文件)           │
│               │  │     └── extracted_text (DuckDB摘要+样本)         │
│               │  └── duckdb_cache (新增)                            │
│               │        ├── duckdb_attachment_id → subtask_contexts  │
│               │        ├── summary (JSON)                           │
│               │        └── source_file_hash (SHA256)                │
│               └─────────────────────┘                                │
│                                                                      │
│  Executor Manager (Docker)                                           │
│       └── ClaudeCode Agent                                           │
│               ├── MCP: wegent-data-analysis                          │
│               │   (wegent_data_schema, wegent_data_query)            │
│               └── DuckDBQueryExecutor                                │
│                   (下载.duckdb + 本地查询执行)                        │
└─────────────────────────────────────────────────────────────────────┘
```

### 2.2 职责分离

| 模块 | 职责 | 说明 |
|------|------|------|
| **knowledge_runtime** | 生成 .duckdb 文件 | 保持无状态，不负责查询执行 |
| **Backend** | MCP 工具 + 元数据管理 | 提供工具接口，从 duckdb_cache 获取 schema |
| **Executor** | 查询执行 + 本地缓存 | 下载 .duckdb 文件，在容器内执行 SQL |

### 2.3 架构设计决策

#### 为什么查询在 Executor 而非 knowledge_runtime

| 对比维度 | knowledge_runtime 查询 | Executor 容器查询 |
|---------|----------------------|------------------|
| 集群部署 | 每个节点需要独立缓存 (N× 存储) | 每个容器独立缓存 |
| 无状态设计 | 缓存破坏无状态 | knowledge_runtime 保持无状态 |
| 缓存生命周期 | 需要 LRU/TTL 管理 | 跟随容器生命周期（自动清理） |
| 内存泄漏风险 | `_locks` 字典可能无限增长 | 容器销毁自动清理 |

#### 为什么对话场景不支持 DuckDB 生成

| 问题 | 说明 |
|------|------|
| 超时设计缺陷 | 同步生成的 5 秒超时对于最大 500MB 文件不可行 |
| 无失败状态 | 超时后无状态记录，用户不知道 SQL 功能不可用 |
| 无重试机制 | 失败后无法重试 |

**解决方案**: 对话场景暂不支持，后续可通过异步消息队列实现。

## 三、数据流程

### 3.1 知识库场景：上传时异步生成

```text
用户上传 Excel/CSV 到知识库
    │
    ▼
POST /api/knowledge/documents
    │
    ▼
1. 存储原始 Excel 文件（现有流程）
   SubtaskContext.binary_data via StorageBackend
    │
    ▼
2. RAG 索引（现有流程）
    │
    ▼
3. 异步生成 DuckDB（Celery 任务，新增）
   ├── Backend 调度 Celery 任务 generate_duckdb_task
   │
   ├── 任务内调用 knowledge_runtime POST /internal/data/generate
   │   ├── ContentFetcher 获取 Excel 文件
   │   ├── 导入 DuckDB（write 连接，允许外部访问）
   │   │   ├── .xlsx → read_xlsx()（DuckDB excel 扩展）
   │   │   └── .csv/.tsv → read_csv_auto()
   │   ├── 执行 SUMMARIZE + 提取样本数据
   │   ├── CHECKPOINT → 确保 WAL 合并到主文件
   │   ├── 关闭 write 连接
   │   │
   │   ├── 上传 .duckdb 文件到 Backend 内部端点
   │   │   POST /api/internal/data/artifacts/upload
   │   │   → ContextService.upload_attachment() → StorageBackend 存储
   │   │   → 返回 duckdb_attachment_id
   │   │
   │   └── 返回 summary + 表信息
   │
   └── Backend 写入 duckdb_cache 表
       (attachment_id, duckdb_attachment_id, summary, source_file_hash, ...)
       + 更新 SubtaskContext.extracted_text 为摘要内容
```

### 3.2 数据查询流程

```text
用户提问："分析各产品销售额 TOP 10"
    │
    ▼
1. AI 从上下文中读取 DuckDB 摘要（替代原 extracted_text 截断文本）
   知道有表 sales_2024，字段 date/product/amount/region
    │
    ▼
2. AI 调用 MCP 工具获取详细表结构
   wegent_data_schema(attachment_id=789)
   → Backend 从 duckdb_cache 表获取 schema
   → 返回 ContentRef + 表结构信息
    │
    ▼
3. AI 生成 SQL
    │
    ▼
4. AI 调用 MCP 工具执行查询
   wegent_data_query(attachment_id=789, sql="SELECT product, SUM(amount)... ")
   → Backend 返回 ContentRef + 表名列表
   → Executor 下载 .duckdb 文件
   → Executor 本地执行 SQL（:memory: + ATTACH 只读模式）
   → 返回查询结果
    │
    ▼
5. AI 呈现结果（表格/图表展示给用户）
```

## 四、存储设计

### 4.1 复用 StorageBackend 架构

.duckdb 文件存储复用现有可插拔 StorageBackend 抽象：

```text
knowledge_runtime → 生成.duckdb → 调用 Backend 内部上传端点
                                    ↓
                        ContextService.upload_attachment()
                                    ↓
                        StorageBackend.save()
                        (当前: MySQL LONGBLOB, 未来可插拔 S3/MinIO)
```

### 4.2 duckdb_cache 表结构

| 字段 | 类型 | 说明 |
|-----|------|------|
| id | INT AUTO_INCREMENT | 主键 |
| attachment_id | INT NOT NULL UNIQUE | 原始附件 ID（关联 subtask_contexts.id） |
| duckdb_attachment_id | INT NULLABLE | .duckdb 文件的附件 ID（关联 subtask_contexts.id） |
| summary | JSON | SUMMARIZE 摘要 + 样本数据 |
| tables_count | INT | 表数量 |
| file_size | INT | .duckdb 文件大小（字节） |
| source_file_hash | VARCHAR(64) | 原始文件 SHA256（完整性校验） |
| status | VARCHAR(20) | generating/ready/failed |
| created_at | TIMESTAMP | 创建时间 |
| updated_at | TIMESTAMP | 更新时间 |

**注意**：
- `duckdb_attachment_id` 允许为 NULL，以便在生成过程中跟踪状态
- 不使用数据库级外键约束，引用完整性由应用层管理

### 4.3 缓存失效策略

| 场景 | 处理方式 |
|------|---------|
| 删除附件 | 应用层同步删除 duckdb_cache 记录 + .duckdb 附件 |
| 重新上传（生成新 attachment_id） | 新的 duckdb_cache 记录，旧记录由旧附件删除时清理 |
| 文件覆盖（同一 attachment_id） | 计算 `source_file_hash`，与缓存中的 hash 比较：匹配则跳过生成，不匹配则重新生成 |

### 4.4 重新索引变更检测

当用户重新索引知识库文档时，系统通过 `source_file_hash` 检测文件是否变更：

```text
用户触发重新索引
    │
    ▼
Backend 读取 duckdb_cache.source_file_hash
    │
    ▼
调用 knowledge_runtime POST /internal/data/generate
├── 请求携带 existing_source_file_hash
    │
    ▼
knowledge_runtime 计算源文件 hash
    │
    ├── hash 匹配 → 返回 source_unchanged=true，跳过生成
    │
    └── hash 不匹配 → 重新生成 DuckDB
```

**优势**：
- 避免重复生成相同文件的 DuckDB
- 减少不必要的计算和存储开销
- 加速重新索引流程

### 4.5 Executor 本地缓存

```text
/tmp/wegent_duckdb_cache/                  # 固定缓存目录
├── {attachment_id_sha256}.duckdb          # 按 attachment_id 哈希命名
└── ...
```

**管理策略**：
- 缓存生命周期跟随容器生命周期
- 容器销毁时自动清理
- 无需 LRU/TTL 管理

## 五、DuckDB 连接管理

### 5.1 生成连接（write 连接）

```python
# 生成阶段：允许外部访问（read_xlsx/read_csv_auto 需要）
conn = duckdb.connect(str(temp_duckdb_path))
conn.execute("SET memory_limit = '4GB'")
conn.execute("SET temp_directory = '/tmp/duckdb_spill'")
conn.execute("SET preserve_insertion_order = false")

# 导入数据
conn.execute("INSTALL excel; LOAD excel;")
conn.execute("CREATE TABLE sales_2024 AS SELECT * FROM read_xlsx(?, header=true)", [excel_path])

# 生成摘要
summary = conn.execute("SUMMARIZE sales_2024").fetchall()

# 确保 WAL 合并
conn.execute("CHECKPOINT")
conn.close()
```

### 5.2 查询连接（:memory: + ATTACH 只读）

```python
# 查询阶段：内存库 + ATTACH 只读，允许创建临时表/视图
conn = duckdb.connect(':memory:')
conn.execute(f"ATTACH '{duckdb_path}' (READ_ONLY) AS data_db")

# 安全配置：禁用外部访问（已 ATTACH 的数据不需要再读外部文件）
conn.execute("SET enable_external_access = false")

# AI 可以创建临时表做复杂分析
conn.execute("CREATE TEMP TABLE intermediate AS SELECT * FROM data_db.sales_2024 WHERE amount > 1000")

# 查询时表名需带前缀 data_db.
result = conn.execute("SELECT * FROM data_db.sales_2024 LIMIT 10").fetchall()
```

**为什么用 :memory: + ATTACH 而非直接 open read_only**：

| 场景 | 直接 read_only | :memory: + ATTACH |
|------|---------------|-------------------|
| SELECT 查询 | ✅ | ✅ |
| 临时表 | ❌ 不支持 | ✅ 在内存库创建 |
| 临时视图 | ❌ 不支持 | ✅ 在内存库创建 |
| 复杂多步分析 | ❌ 受限 | ✅ 完整支持 |
| enable_external_access=false | ✅ 安全 | ✅ 安全 |

## 六、安全控制

### 6.1 分层安全策略

| 连接类型 | read_only | enable_external_access | 说明 |
|---------|-----------|----------------------|------|
| 生成连接 | false | true（默认） | 需要 read_xlsx/read_csv_auto |
| 查询连接 | ATTACH 只读 | false | 已 ATTACH 的数据无需再读外部文件 |

### 6.2 连接级安全

**安全措施**：

```python
# 1. ATTACH with READ_ONLY: 阻止所有写操作
conn.execute(f"ATTACH '{duckdb_path}' AS data_db (READ_ONLY)")

# 2. 禁用外部访问: 阻止文件/网络访问
conn.execute("SET enable_external_access = false")
```

**安全效果**：

| 攻击向量 | READ_ONLY | enable_external_access=false |
|---------|-----------|------------------------------|
| DROP/DELETE/INSERT/UPDATE | ✅ 阻止 | - |
| read_csv_auto/read_xlsx | - | ✅ 阻止 |
| read_text/read_blob | - | ✅ 阻止 |
| CREATE TABLE | ✅ 阻止 | - |
| CREATE TEMP TABLE | ✅ 允许 | - |

### 6.3 其他防护措施

| 措施 | 说明 |
|------|------|
| 返回行数限制 | 默认 5000 行，超限截断并标记 `truncated=true` |
| 文件大小限制 | `DUCKDB_MAX_FILE_SIZE_MB`（默认 500MB） |
| 权限校验 | MCP 工具通过 `token_info` 验证用户对附件的访问权限 |

### 6.4 内存保护

| 措施 | 说明 |
|------|------|
| 生成前检查 | 检测系统可用内存，不足则拒绝（`DUCKDB_MIN_FREE_MEMORY_MB`，默认 1024MB） |
| memory_limit | 生成连接设置 `memory_limit='4GB'`，DuckDB 自动 spill to disk |
| temp_directory | 配置临时目录用于内存溢出时的磁盘 spill |

**功能**：返回 ContentRef 供 Executor 下载并执行 SQL

**输出**：

```json

  "content_ref": {...},
  "tables": ["sales_2024"],
  "sql": "SELECT product, SUM(amount) FROM data_db.sales_2024 GROUP BY product",

```

```text

    ▼

    │
    ▼

    │
    ▼

    ├── SET enable_external_access = false
    └── 执行查询，返回结果
```

## 九、AI 上下文注入

### 9.1 extracted_text 新格式

当 DuckDB 生成成功后，`SubtaskContext.extracted_text` 被替换为：

```markdown
## Data Summary: sales_2024.xlsx

### Table: sales_2024 (1,000,000 rows)

| Column | Type | Min | Max | Unique | Avg | Null% |
|--------|------|-----|-----|--------|-----|-------|
| date | DATE | 2024-01-01 | 2024-12-31 | 366 | - | 0% |
| product | VARCHAR | - | - | 150 | - | 0% |
| amount | DOUBLE | 0.01 | 99999.99 | - | 1523.45 | 2% |
| region | VARCHAR | - | - | 8 | - | 5% |

> Use wegent_data_schema to get full table structure, wegent_data_query to execute SQL queries for detailed analysis.
```

### 9.2 降级策略

```text
Excel/CSV 上传到知识库
    │
    ▼
尝试 DuckDB 生成（Celery 异步任务）
    │
    ├── 成功 → extracted_text = 摘要 + 样本
    │
    └── 失败（内存不足/格式异常/超时）
        ├── duckdb_cache.status = 'failed'
        └── extracted_text 保持原有文本提取内容
```

## 十四、已知限制与未来规划

### 14.1 当前限制

| 限制 | 说明 |
|------|------|
| 对话场景不支持 | 同步生成超时设计有缺陷，需重新设计异步方案 |
| 查询超时未显式配置 | 依赖 DuckDB 内部超时机制 |

### 14.2 未来规划

| 功能 | 优先级 | 说明 |
|------|--------|------|
| 对话场景异步生成 | 中 | 通过消息队列实现异步 DuckDB 生成 |
| 查询超时配置 | 高 | 添加 `statement_timeout` 配置 |
