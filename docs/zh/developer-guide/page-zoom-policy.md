# 页面缩放策略

## 背景

Wegent 的移动端和桌面客户端应保持稳定的页面布局，不允许用户通过浏览器或 WebView 的默认能力手动缩放整个页面。同时，图片预览、画布、远程桌面等业务组件仍可提供自己的缩放控件。

当前 Next.js 前端已在根布局中设置固定 viewport，但独立的 `wework` 客户端仅声明了初始缩放比例，也没有统一处理桌面端页面缩放快捷键。

## 目标

- 移动端禁止通过双指手势缩放整个页面。
- 桌面端禁止通过 `Ctrl/Cmd + +/-/0` 缩放或重置整个页面。
- 桌面端禁止通过 `Ctrl/Cmd + 滚轮` 缩放整个页面。
- 在支持原生手势事件的 WebView 中禁止页面级缩放手势。
- 保留业务组件通过按钮、滑杆或组件内部事件实现的缩放。

## 非目标

- 不移除图片、画布、图表或远程桌面的缩放按钮。
- 不拦截没有修饰键的滚动。
- 不改变操作系统的显示缩放或辅助功能设置。
- 不通过 Tauri Rust 层修改 WebView 的渲染比例。

## 方案

### 声明式 viewport

`wework/index.html` 使用与 Next.js 前端一致的 viewport 策略：

```html
<meta
  name="viewport"
  content="width=device-width, initial-scale=1, maximum-scale=1, user-scalable=no"
/>
```

该配置负责移动浏览器和移动 WebView 的页面缩放限制。Next.js 前端继续使用根布局中现有的 `maximumScale: 1` 和 `userScalable: false` 配置。

### 集中式页面缩放守卫

在 `wework` 中新增独立的页面缩放守卫模块，并由应用入口安装。该模块只处理会触发浏览器或 WebView 默认页面缩放的输入：

- `keydown`：当 `Ctrl` 或 `Meta` 被按下时，拦截 `+`、`-`、`=`、数字小键盘加减键和 `0`。
- `wheel`：当 `Ctrl` 或 `Meta` 被按下时阻止默认行为。
- `gesturestart` 和 `gesturechange`：阻止支持这些事件的平台执行原生页面缩放。

守卫不处理普通点击、组件状态或不带修饰键的滚轮事件。业务组件的显式缩放按钮和内部缩放逻辑因此不受影响。

安装函数返回清理函数，以便测试和未来的应用卸载场景移除所有监听器。事件监听使用满足浏览器要求的选项；需要调用 `preventDefault()` 的监听器不得设置为 passive。

## 数据流

1. `wework` 应用入口启动。
2. 页面缩放守卫在 `document` 上注册输入事件监听器。
3. 用户触发页面级缩放输入时，守卫调用 `preventDefault()`。
4. 其他输入继续传播到业务组件。
5. 应用卸载或测试结束时调用清理函数移除监听器。

该功能不访问后端、不持久化状态，也不引入新的用户配置。

## 测试

单元测试应覆盖：

- 阻止 `Ctrl` 和 `Meta` 组合的放大、缩小与重置快捷键。
- 阻止带 `Ctrl` 或 `Meta` 的滚轮事件。
- 阻止原生缩放手势事件。
- 不阻止普通按键和普通滚轮事件。
- 清理函数移除所有事件监听器。
- `wework` viewport 包含固定缩放声明。

验证时运行 `wework` 的相关单元测试、TypeScript 检查和 lint。现有业务组件缩放测试应继续通过。

## 风险与约束

浏览器可能允许用户通过开发者工具或浏览器菜单改变缩放，这不属于页面脚本能够可靠控制的范围。Tauri 客户端没有暴露缩放菜单，因此本方案覆盖客户端正常可用的手动输入路径。

页面缩放限制会降低浏览器缩放这一辅助功能的可用性。业务 UI 必须继续满足可读性和响应式要求，操作系统级辅助功能不应被禁用。
