# 技能启用与临时追加产品设计

## 背景

当前技能交互把“上传技能”“选择智能体”“聊天时选择技能”混在一起。用户上传技能后，系统默认智能体不能直接使用，必须新建或编辑智能体才能获得类似体验；但默认智能体本身已有系统内置技能，用户很难复刻一个完整的“带技能智能体”。

新的主心智采用：

> 技能默认属于“我”，不是属于某个智能体。我的技能启用后，会在我使用助手时自动可用；聊天里仍可临时追加几个技能做测试。

这比“上传后必须配置生效范围”更符合小白用户对“上传了就能用”的预期，同时通过兼容规则避免影响现有系统行为。

## 核心目标

- 新上传或新导入的技能默认可用，避免“上传成功但用不了”。
- 现有存量技能默认不启用，避免上线后改变已有任务、默认助手和系统运行状态。
- 聊天输入框保留临时追加逻辑，用于本次测试或偶发使用。
- 技能启用状态只控制“我的全局技能”，不改变系统助手、共享助手或智能体内置技能。
- 高级用户可以进一步设置例外，但普通用户不需要先理解智能体、Ghost、Shell 或内置技能。

## 非目标

- 不要求用户为了使用技能创建智能体。
- 不要求用户复制系统默认智能体。
- 不把“我的技能启用状态”写入 `Ghost.spec.skills`。
- 不取消系统或智能体已有的内置技能。
- 第一版不把“指定场景 / 指定智能体生效”作为默认配置流程。

## 用户心智

```text
我的技能
  我上传、导入或安装的能力资产

启用
  这个技能会自动参与我的对话

未启用
  这个技能在我的技能库里，但不会自动参与对话
  仍然可以在聊天里本次临时追加

临时追加
  只对当前这一次发送生效
  不改变技能启用状态
```

一句话：

> 我的技能默认跟着我走；启用后自动可用；没启用的技能也可以在聊天中临时试用。

## 启用状态

技能列表里每个技能有一个明确状态：

```text
已启用
未启用
```

状态含义：

- `已启用`：默认对我的所有对话可用，包括默认助手、系统助手、共享助手和我创建的助手。
- `未启用`：不自动参与对话，但可以在聊天技能按钮中临时追加。

重要边界：

- 启用状态是当前用户自己的配置，不影响其他用户。
- 启用状态不修改智能体本体，不修改系统助手，不修改共享助手。
- 如果某个技能已经是智能体内置技能，它仍然按智能体内置逻辑生效；“我的技能启用状态”只控制额外叠加的个人技能。

## 兼容规则

为避免影响现有系统：

```text
存量技能：默认未启用
新上传技能：默认启用
新导入技能：默认启用
系统/公共技能：不因为本功能自动变成“我的已启用技能”
智能体内置技能：保持原有行为
```

这能保证上线后：

- 过去已经上传但没有绑定到智能体的技能，不会突然进入所有对话。
- 系统默认助手已有能力不变。
- 用户从新版本开始上传的技能，符合“上传了就能用”的预期。

## 上传后交互

上传成功后不再要求用户选择复杂生效范围。

上传成功反馈只负责确认结果，不承担唯一教学职责。反馈内容必须和后续“我的技能”页里的常驻状态一致：

```text
技能已上传，已为你启用。
之后对话中会自动可用。

[查看我的技能]
```

如果用户从高级入口上传，也可以提供次级入口：

```text
已启用
你可以稍后在“我的技能”里关闭，或添加例外。
```

这里必须解释“启用”的含义，避免用户误以为技能被写进了某个智能体：

```text
已启用表示：你使用任何助手时，这个技能都会自动可用。
不会修改助手本身，也不会影响其他用户。
```

用户关闭反馈后，仍然能在“我的技能”列表、技能详情和聊天技能按钮里看到同样的含义说明。

## 我的技能页

列表展示：

```text
技能名称
描述
状态：已启用 · 会在你的对话中自动可用
状态：未启用 · 不会自动使用，可在聊天时临时追加
来源：上传 / 导入 / 系统 / 资源库
操作：启用 / 关闭 / 详情
```

详情页展示：

```text
启用状态
[开关] 自动在我的对话中可用

高级设置
例外规则
```

第一版高级设置可以先只保留入口，不一定一次性实现完整例外系统。

列表顶部需要有一个长期存在的短说明：

```text
已启用的技能会在你的对话中自动可用；未启用的技能也可以在聊天时临时追加。
```

列表中的状态标签不能只显示“开 / 关”，需要同时解释结果：

```text
已启用 · 自动可用
未启用 · 可临时追加
```

空状态需要直接给出下一步：

```text
还没有我的技能
上传或导入一个技能后，它会默认启用，并在你的对话中自动可用。
```

## 例外规则

例外是高级能力，不是默认路径。

产品语义：

```text
默认：启用后在我的所有对话可用
例外：我不希望它在某些地方自动出现
```

建议分阶段：

第一版最小能力：

- 暂停技能：等同于关闭启用状态。

第二版可扩展：

- 不在某些助手中自动使用。
- 不在某些场景中自动使用。

不建议第一版就把“所有助手 / 指定场景 / 指定助手”作为正向配置主流程，否则会回到“上传后还要配置才能用”的复杂心智。

## 聊天技能按钮

聊天输入框保留技能按钮，但定位是：

```text
查看自动可用技能 + 本次临时追加技能
```

打开后分两块：

```text
自动可用
  已启用的我的技能，会自动参与当前对话
  只读展示，不在这里做长期配置

本次临时追加
  只影响这条消息，发送后不会启用
  可以勾选未启用或未自动进入当前对话的技能
```

底部入口：

```text
管理我的技能
```

技能按钮内需要有长期存在的结构化说明，而不是依赖独立的新手教程：

```text
自动可用
已启用的技能会自动参与当前对话，不需要每次选择。

本次临时追加
只影响这条消息，适合临时试用未启用技能。
```

当“自动可用”为空时，展示：

```text
暂无自动可用技能
你可以在“我的技能”中启用技能，或在下方为本次临时追加。
```

当“本次临时追加”为空时，展示：

```text
没有可临时追加的技能
```

当用户多次临时追加同一个技能，可以提示：

```text
经常使用这个技能？启用后以后会自动可用。
[启用]
```

这个动作可以直接启用，因为语义很明确：从“临时试用”升级为“自动可用”。

临时追加后，输入框上方应该持续显示本次追加状态，而不是只弹一次提示：

```text
本次临时追加：Markdown 表格整理
只影响这条消息，发送后不会自动启用。
```

如果展示成标签，标签也要带有临时语义：

```text
Markdown 表格整理 · 本次
```

## 聊天页展示

聊天页不要让技能和智能体成为同级主选择。主选择仍然是智能体，技能只是能力说明。

建议摘要：

```text
当前助手：默认助手
技能：自动可用 3 个，临时追加 1 个
```

技能详情中按来源展示：

```text
助手自带
我的技能
本次临时追加
```

去重规则：

- 同一个技能如果既是助手自带，又是我的已启用技能，只展示一次。
- 展示优先级：助手自带 > 我的技能 > 本次临时追加。

## 与智能体内置技能的关系

产品上必须区分两类能力：

```text
助手自带能力
  是智能体定义的一部分
  跟着智能体分享或发布

我的技能
  属于当前用户
  默认跟着当前用户走
  不改变智能体本体
```

普通用户默认只需要理解“我的技能启用后自动可用”。

高级用户在智能体编辑页看到的“内置能力”应使用不同文案，避免和“我的技能启用状态”混淆。

## 产品规则

1. 新上传或新导入技能默认启用。
2. 存量技能默认未启用，保护现有系统行为。
3. 启用状态只控制当前用户的全局可用技能。
4. 聊天里的临时追加只影响本次发送。
5. 临时追加不改变启用状态，除非用户明确点击“启用”。
6. 智能体内置技能继续按原有逻辑生效。
7. 技能重复命中时只展示和下发一次。

## 推荐文案

- 使用“启用”，不用“生效范围”作为主入口。
- 使用“自动可用”，解释启用后的效果。
- 使用“本次临时追加”，解释聊天里的勾选行为。
- 使用“关闭启用”，不要说“解绑智能体”。
- 使用“助手自带能力”和“我的技能”，区分智能体内置和个人技能。

## 长期界面引导

这套设计的引导是长期界面能力，不是一次性教程或 toast。用户每次看到技能相关界面时，都应该能从界面结构和操作文案理解：

- 哪些技能已经自动可用。
- 哪些技能只是本次临时追加。
- 启用状态属于“我的技能”，不是修改助手本体。
- 助手自带能力和我的技能不是一回事。

长期引导由以下界面元素承担：

- 分区标题：`自动可用`、`本次临时追加`、`助手自带`、`我的技能`。
- 状态标签：`已启用`、`未启用`、`本次追加`、`助手自带`。
- 空态说明：告诉用户为什么为空以及下一步能做什么。
- 操作文案：`启用后自动可用`、`本次临时追加`、`关闭自动可用`。
- 来源标签：解释技能来自助手内置、我的技能，还是本次临时选择。

### 上传成功

目的：确认新技能已经可用，但不把它当成唯一引导。用户关闭反馈后，后续界面仍然持续展示“已启用”和“自动可用”的含义。

```text
技能已上传，已为你启用。
之后使用任何助手时都会自动可用。
```

次级说明：

```text
启用只影响你的对话，不会修改助手本身。
```

### 我的技能列表

目的：长期解释启用状态和临时追加的区别。

```text
已启用的技能会自动可用；未启用的技能不会自动参与，但可以在聊天时临时追加。
```

每个技能行都要把状态和结果绑定展示：

```text
已启用 · 会在你的对话中自动可用
未启用 · 不会自动使用，可在聊天时临时追加
```

### 聊天技能按钮

目的：长期解释为什么有些技能不能勾选。

```text
自动可用技能已经会参与当前对话，不需要勾选。
下方技能只会本次临时追加。
```

### 临时追加后

目的：在本次消息发送前持续解释临时追加不会改变长期状态。

```text
本次临时追加只影响这一条消息。经常使用的话，可以启用为自动可用。
```

### 智能体编辑页

目的：长期避免“助手自带能力”和“我的技能”混淆。

```text
助手自带能力会跟随这个助手；我的技能只影响你自己的对话。
```

## 第一版范围

第一版只做：

- 我的技能增加启用状态。
- 存量技能默认未启用。
- 新上传 / 新导入技能默认启用。
- 聊天技能按钮拆成“自动可用”和“本次临时追加”。
- 临时追加继续走现有追加逻辑。
- 通过状态、副标题、空态、来源标签和临时追加标签提供持续解释。

第一版不做完整例外规则；只保留后续可扩展空间。
