# 技能界面重规划产品设计

## 背景

技能系统需要同时解释三件事：

- Skill 是一个资产，可能来自个人、团队、系统或资源库。
- 用户希望“上传了就能用”，但不是所有 Skill 都应该在所有场景里生效。
- 高级用户需要控制某些自动启用 Skill 在指定场景里不要出现。

之前的界面容易把“技能来源”“是否自动启用”“绑定到智能体”混在一起。新的界面把主心智收敛为：

> 技能库管理我能使用的 Skill；自动启用技能管理哪些 Skill 会自动加入我的对话；例外管理某个自动启用 Skill 在哪些场景里不要自动加入。

## 设计目标

- 小白用户能一眼理解：哪些技能现在会自动跟着我。
- 上传或添加 Skill 后，能自然进入“自动启用技能”流程。
- 团队 Skill、系统 Skill 可以被当前用户自动启用，但不会被误解为“归我所有”。
- 例外能力存在，但不干扰主界面的简单心智。
- 高级设置页不呈现为规则表，而是按 Skill 管理例外。
- 为未来项目级例外保留扩展位置。

## 非目标

- 不在第一层界面展示复杂的启用范围配置。
- 不做规则表式的后台管理界面。
- 不把某个用户的自动启用配置写回 Skill 资产。
- 不让某个用户启用团队 Skill 后影响其他团队成员。
- 不在本次设计里实现项目级能力，只保留界面扩展点。

## 命名

推荐将“默认启用技能”改名为 **自动启用技能**。

原因：

- “默认启用”容易让用户联想到系统默认、全局默认或管理员配置。
- “自动启用”表达的是用户可理解的行为：这些技能会自动加入我的对话。
- 例外也更容易解释：这个 Skill 默认会自动启用，但在某些场景不自动启用。

## 一级页：技能

一级页保留两个主要区域。

### 自动启用技能

位置在技能页面顶部，是主心智区域。

用途：

```text
告诉用户：哪些 Skill 会自动加入我的对话。
```

每个条目展示：

- Skill 名称。
- 来源：我的技能 / 团队技能 / 系统技能 / 资源库技能。
- 自动启用状态：
  - `所有对话自动启用`
  - `有 N 个例外`
- 例外摘要，例如：
  - `编码模式不启用`
  - `财务分析智能体不启用`

主要操作：

- `添加技能`：从当前用户可用的所有 Skill 中选择，加入自动启用技能。
- `取消自动启用`：移出自动启用技能，但不删除 Skill 资产。
- `高级设置`：进入自动启用设置页，管理例外。

空状态：

```text
还没有自动启用技能
添加后，这些技能会自动加入你的对话。

[添加技能]
```

### 技能库

位置在自动启用技能下方，是资产管理区域。

用途：

```text
管理我能使用的 Skill 资产，并可把常用 Skill 设为自动启用。
```

来源筛选：

- `全部`
- `我创建的`
- `团队`
  - 下拉支持 `全部团队`
  - 下拉支持选择单个团队
- `系统`

技能卡片展示：

- Skill 名称。
- 描述。
- 来源。
- 所属团队，若是团队 Skill。
- 是否已自动启用。
- 例外数量，若已自动启用且存在例外。
- 上传、更新、下载、删除等资产操作。
- `设为自动启用` / `取消自动启用`。

这里的来源筛选只影响技能库列表，不影响自动启用技能区域。

## 二级页：自动启用设置

入口来自一级页自动启用技能区域的 `高级设置`。

页面不使用规则表，不使用多 tab 作为主结构。主结构是：

```text
自动启用 Skill 列表
  点击某个 Skill
    打开右侧抽屉
      管理这个 Skill 的例外
```

### 页面头部

标题：

```text
自动启用设置
```

说明：

```text
管理每个自动启用技能的例外。
```

操作：

- `添加自动启用技能`

### 自动启用 Skill 列表

列表只展示已经自动启用的 Skill。

每行展示：

- Skill 名称。
- 来源。
- 状态：
  - `所有对话自动启用`
  - `有 N 个例外`
- 例外摘要。
- 点击进入详情抽屉。

示例：

```text
translate-zh-en
我的技能 · 所有对话自动启用

交互式表单提问
系统技能 · 有 1 个例外：编码模式不启用
```

### Skill 详情抽屉

点击某个自动启用 Skill 后打开。

抽屉内容：

- Skill 名称和来源。
- 默认行为：
  - `所有对话自动启用`
- 不启用的模式：
  - 聊天
  - 编码
  - 设备
  - 其他现有模式
- 不启用的智能体：
  - 可选择系统智能体、共享智能体、自己创建的智能体。
  - 不要求用户拥有智能体编辑权限，因为这是当前用户自己的例外配置。
- 未来扩展：
  - 不启用的项目。

操作：

- 添加例外。
- 删除例外。
- 取消此 Skill 的自动启用。

### 检查场景

高级设置页可以提供一个辅助工具，但不作为主入口。

用途：

```text
选择一个智能体和模式，查看这个场景最终会自动启用哪些 Skill。
```

输入：

- 智能体。
- 模式。

输出：

- 会自动启用的 Skill。
- 因例外被排除的 Skill。

这个工具用于解释和排查，不用于创建主配置。

## 关键心智

### Skill 资产归属

Skill 资产仍然有来源：

- 我的技能。
- 团队技能。
- 系统技能。
- 资源库技能。

来源回答：

```text
这个 Skill 从哪里来，谁能管理它。
```

### 自动启用归属

自动启用配置属于当前用户。

它回答：

```text
这个 Skill 是否会自动加入我的对话。
```

因此：

- 团队 Skill 可以被我自动启用。
- 这不会改变团队 Skill 资产。
- 这不会影响团队其他成员。
- 系统 Skill 可以被我自动启用。
- 这不会修改系统智能体或系统配置。

### 例外归属

例外也属于当前用户。

它回答：

```text
这个已经自动启用的 Skill，在哪些场景里不要自动加入我的对话。
```

例外不是修改智能体，也不是修改 Skill 资产。

## 聊天界面关系

聊天输入框里的 Skill 弹层应展示三类信息：

- 自动启用技能：
  - 只读展示。
  - 可提供 `本次不使用`。
- 智能体自带技能：
  - 只读展示。
  - 不允许临时移除。
- 本次临时使用：
  - 用户可临时追加 Skill。
  - 发送后不改变长期配置。

如果用户想长期调整自动启用，应引导到：

```text
自动启用设置
```

## 数据模型建议

继续沿用使用位置绑定模型，但增加例外概念。

```text
SkillBinding
  targetType: user
  targetId: currentUser
  skillRef

SkillBindingException
  bindingId
  exceptionType: mode | agent | project
  exceptionValue
```

解释：

- `SkillBinding targetType=user` 表示当前用户自动启用某个 Skill。
- `SkillBindingException` 表示这个自动启用绑定在某些场景下不生效。
- 第一版实现 `mode` 和 `agent`。
- `project` 作为未来扩展。

## 兼容规则

- 存量 Skill 不自动加入任何用户的自动启用技能。
- 新上传个人 Skill 默认加入上传者的自动启用技能。
- 新上传团队 Skill 默认只加入上传者的自动启用技能。
- 系统 Skill 不自动加入用户的自动启用技能，可由用户手动添加。
- 现有智能体自带 Skill 继续按智能体生效，不迁移到用户自动启用。
- 聊天临时选择 Skill 继续只影响本次发送。

## 验收标准

- 用户进入技能页，能明确看到哪些 Skill 会自动进入自己的对话。
- 用户能从技能库把任何可用 Skill 加入自动启用技能。
- 用户能从自动启用技能区域取消自动启用。
- 用户能进入高级设置，按 Skill 配置模式例外和智能体例外。
- 团队 Skill 被自动启用时，界面明确提示只影响当前用户。
- 来源筛选只影响技能库，不影响自动启用技能区域。
- 聊天界面能解释自动启用、智能体自带、本次临时使用三类 Skill 的来源。
