# Wework Codex 风格工作台 MVP 设计

## 背景

`wework/` 当前是独立的 Tauri 2 + Vite + React + TypeScript 脚手架，只有基础 `AppLayout`、`Header`、`Sidebar` 和 `HomePage`。本阶段目标不是做静态 UI 原型，而是把 `wework` 做成真实可用的多端客户端 UI：PC/Web 参考 Codex 桌面工作台的信息架构，移动端参考 ChatGPT 移动首页和侧栏体验。

业务后端不重建。`wework` 复用现有 Wegent Backend API、WebSocket/SSE、Task、Project、Team、Message 等数据结构，作为后续“编码功能迁移”的第一阶段。

## 目标

- PC/Web 端提供 Codex-like 工作台：左侧项目与历史会话，右侧聊天/任务工作区。
- 移动端提供首页输入体验和抽屉式侧栏：底部输入框、快捷入口、项目与历史会话。
- 接入真实项目列表、历史会话、聊天任务创建、消息发送和流式回复。
- 保持现有 Wegent 消息数据原则：统一消息状态是 UI 的唯一渲染来源。
- 建立清晰客户端边界，方便后续迁移更完整的编码执行功能。

## 非目标

- 不迁移完整代码编辑、终端、文件树、diff、设备执行面板。
- 不新建独立后端或新的业务数据模型。
- 不把现有 `frontend` 的复杂页面整体搬进 `wework`。
- 不在 MVP 中实现图片生成、语音输入等高级能力；相关入口在第一阶段只作为不可执行的导航项展示。

## 推荐方案

采用“外壳优先 MVP”：

1. 先搭建响应式 Codex-like 工作台外壳。
2. 再通过小而清晰的客户端适配层接入真实数据。
3. 只复用现有业务语义和 API，不复用旧页面布局假设。

该方案能最快形成可试用产品，同时为后续编码功能迁移留下清晰扩展点。

## 架构

`wework/src` 下按职责拆分：

- `api/`：封装 Backend REST API，例如当前用户、项目列表、任务历史、任务详情、创建任务。
- `stream/`：封装 WebSocket/SSE 流式消息订阅，统一处理 `start`、`chunk`、`done`、`error`。
- `features/workbench/`：工作台业务状态，包括当前项目、当前任务、最近会话、输入状态、加载状态、发送状态。
- `components/layout/`：端侧布局组件。PC/Web 和移动端分离实现，避免移动端被桌面侧栏结构绑死。
- `components/chat/`：消息列表、输入框、快捷操作、模型/语言/项目选择等聊天组件。
- `types/`：客户端视图模型类型，只做 Backend 数据的展示层整理，不改变业务语义。

`wework` 作为新客户端，不直接依赖 `frontend` 的页面组件。可参考现有 hooks/service 的行为，但要通过本应用内 adapter 固化边界。

## PC/Web 端设计

PC/Web 端参考第一张图：

- 左侧固定侧栏，宽度约 280px。
- 顶部主操作包含：新对话、搜索、能力/插件入口、自动化入口。
- 项目区展示真实 Project/Workspace 列表，点击后切换当前项目上下文。
- 历史会话区展示最近 Task/Chat，点击后进入对应会话。
- 底部保留设置入口。

主工作区有两种状态：

- 空状态：居中标题“我们该做什么？”，下方是主输入框。输入后创建新任务并进入会话。
- 会话状态：显示真实消息列表，底部固定输入框，支持发送消息和流式展示回复。

PC/Web 端优先服务“选择项目 → 发起任务 → 继续历史会话”的工作流，不做传统 dashboard。

## 移动端设计

移动端参考第二、三张图，分为首页和侧栏：

- 首页顶部：菜单按钮、品牌/模式名、新会话或账户入口。
- 首页主体：保留大量空白，聚焦底部输入框。
- 快捷入口：输入框上方放 2-3 个入口，MVP 使用“新任务”“项目工作”“查找资料”。
- 底部输入框：固定在底部，包含加号、文本输入、语音入口图标和发送按钮，触控目标不小于 44px。
- 侧栏：展示 Wework 标题、搜索、用户头像、主导航项、最近历史会话和醒目的“聊天/新任务”按钮。

项目和历史会话均接真实数据。点击项目或会话后关闭侧栏，并切换到对应上下文。

## 数据流

启动流程：

1. 读取当前登录用户。
2. 读取默认 Team。
3. 并行读取项目列表和最近 Task/Chat 历史。
4. 初始化 workbench 状态。

新建会话：

1. 用户在空状态或移动首页输入消息。
2. 使用当前 Team 和当前项目上下文创建 Task。
3. 将用户消息加入统一消息状态。
4. 发送消息到 Backend。
5. 订阅流式消息并更新 AI 消息状态。

打开历史会话：

1. 用户点击历史 Task。
2. 拉取 Task 详情和历史消息。
3. 将历史消息同步到统一消息状态。
4. 订阅该 Task 的后续流式更新。

消息显示规则：

- UI 只渲染统一消息状态。
- Backend Task 详情中的 subtasks 仅作为初始化来源，不作为持续显示源。
- WebSocket/SSE 是新消息和流式更新的实时来源。

## 错误处理

MVP 需要覆盖以下错误：

- 未登录或会话过期：跳转登录或显示重新登录提示。
- 项目加载失败：侧栏项目区显示重试入口。
- 历史会话加载失败：历史区显示轻量错误和重试入口。
- 创建任务失败：保留用户输入，提示重试。
- 发送消息失败：消息进入失败状态，支持重发。
- 流式连接断开：显示连接状态，可自动重连一次，失败后提供手动重试。

错误 UI 使用 toast 或局部内联提示，不使用大面积错误页。

## 测试策略

- API adapter 单元测试：覆盖成功、失败、认证错误和响应格式转换。
- Stream adapter 单元测试：覆盖 `start`、`chunk`、`done`、`error` 顺序和异常事件。
- Workbench 状态测试：覆盖项目切换、新建会话、打开历史会话、发送失败重试。
- PC/Web 组件测试：验证侧栏项目、历史会话、空状态输入框和会话输入框。
- 移动端组件测试：验证抽屉打开关闭、快捷入口、历史点击、44px 触控要求。
- 集成/E2E 测试：至少覆盖一条真实请求链路方向，不用前端 mock 后端 API 来让测试静默通过。

## 后续扩展

MVP 完成后，第二阶段再迁移编码相关能力：

- 项目文件树和工作区详情。
- 代码任务执行状态。
- 终端/设备任务入口。
- diff、产物、日志和运行结果面板。
- 更完整的 Codex-like 项目工作流。
